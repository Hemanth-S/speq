# PRD: Speq Improvements

## Context

[speq](speq.md) is a spec-driven TDD workflow for Claude Code — 8 slash commands that drive PRD → specs → tasks → TDD implementation → verification → done. Today it runs as a single Claude Code session, with prompts baked into each command, no observability beyond `bd list`, and no machine-readable architecture constraints.

This PRD captures the next wave of improvements, drawn from patterns observed at AI Tinkerers events: [agent-teams](https://github.com/sransom/agent-teams) (parallel worktree fan-out, model routing), actual.ai's ADR-as-guardrail pattern, [thrum](https://leonletto.github.io/thrum/) (durable agent messaging), and the broader move toward eval-driven prompt engineering.

The unifying frame: make speq's pipeline **modular, evidence-based, and observable**, so that as models improve any step, swaps are mechanical and provable rather than vibes-based.

## Goals

- **Modular pipeline.** Each step is a contract with a pluggable implementation, selectable via config. Prompts, models, and implementations can be swapped or elided independently.
- **Evidence-based swaps.** Step-level evals plus cost telemetry let every prompt/model/implementation change be defended with data.
- **Cost transparency.** Per-run cost data, predictive estimates, optional budget guardrails.
- **Architecture durability.** ADRs as machine-readable rubrics that the pipeline enforces from `/requirements` through `/verify`.
- **Observability.** A live sprint board surfaces run state, configuration, and cost.
- **Optional parallelism.** Opt-in multi-agent `/implement` for fan-out features via agent-teams composition.
- **Optional resilience.** Durable orchestrator state that survives session restarts.

## Non-goals

- Multi-user, shared backend, or team-collaboration features. **Single-user, single-machine assumption.**
- Async stakeholder review on PRDs and specs (comment.io integration — deferred; see below).
- Runtime-agnostic *orchestration*. Claude Code remains the outer runtime that drives the pipeline; modularity is within the pipeline. Individual steps may invoke other tools (e.g., the codex CLI) as subprocesses, but the orchestrator is always Claude Code.
- Hosted dashboards or cloud services.
- Real-time multi-machine sync.

## Personas

Solo developer (the speq author) using speq across personal projects on a single machine.

## Constraints

- Claude Code is the *orchestration* runtime; slash commands are the surface. Steps may shell out to other CLIs as subprocesses when configured (e.g., `codex exec` for gpt-* implementer runs).
- Beads remains the task tracker.
- Backward compatible with existing speq installs — features degrade gracefully when their config is absent.
- All existing speq non-negotiables remain in force:
    - Codebase scanned before `/requirements` or `/enrich`.
    - No task closed without GIVEN/WHEN/THEN verified.
    - No code before a failing test.
    - Every spec covers security, failure, idempotency, performance. Security is P0.
    - Inputs validated; secrets from env only; errors never leak stack traces or paths.

**Resolved: Caveman mode split.** Caveman mode is two things: a toggle (pipeline behavior) and instruction text (Claude reads it). ## Features

### F1: Modular pipeline (foundation)

Make each step in speq's pipeline a **contract** — typed inputs, typed outputs — with a pluggable implementation behind it. Add `speq.config.yaml` at the repo root that selects per-step implementation, prompt version, and model. Support three swap modes:

- **Prompt versioning.** Each step has a directory at `prompts/<step>/` containing one or more version files (`v1.md`, `v2.md`, ...). The active version is selected in `speq.config.yaml` (`steps.<name>.prompt_version`). A `speq sync-prompts` preflight (run automatically by `/ship` and `/project:config`) copies the configured version into `.claude/commands/<step>.md`, which is what Claude Code actually reads. Adding a new version is just dropping a new file; F3 step evals run on the swap before it becomes default.
**Resolved: init calls sync-prompts.** `speq init` writes a default `speq.config.yaml` (pointing each step at `prompts/<step>/v1.md`), then runs `sync-prompts` to materialize `.claude/commands/` from the configured versions — same code path `/ship` uses. No confusing intermediate state where prompts shift between init and first run. This is also where init runs the `CLAUDE.md` marker reconciliation for caveman mode.
- **Implementation swap** (e.g., the OpenSpec step run by a script instead of a prompt).
- **Step elision** via capability flags (e.g., skip `/verify` when the configured model declares the `verify-native` capability).

**Configuration coexistence with `CLAUDE.md`.** `CLAUDE.md` keeps everything that is *instructions to Claude* about the project — style, conventions, project facts, caveman, etc. `speq.config.yaml` owns *pipeline knobs* — per-step model/impl/prompt-version, capability flags, cost thresholds. If today's `<!-- BEGIN SPEQ -->` block in `CLAUDE.md` contains pipeline-shaped config, `/project:config migrate` lifts the pipeline keys into `speq.config.yaml` and leaves non-pipeline content in `CLAUDE.md`. Idempotent; safe to re-run.

**Configuration sources, in precedence order:**

1. Well-known top-level CLI flags (`--from=<phase>`, `--max-cost=$X`, `--dry-run`, `--resume`, `--profile=<name>`) and the generic `--set <dotted.path>=<value>` (repeatable) for arbitrary overrides. Example: `/ship --set steps.implement.model=sonnet --set steps.spec.prompt_version=v2`.
2. Env vars using the same dotted-path, uppercased with underscores: `SPEQ_STEPS_IMPLEMENT_MODEL=sonnet`.
3. `speq.config.yaml` at repo root.
4. Shipped defaults.

`/project:config` is an interactive editor that walks current settings, validates edits against the schema, and writes the file.

**Acceptance:**

- GIVEN a project with `speq.config.yaml`, WHEN `/ship` runs, THEN each step reads its impl/version/model from config.
- GIVEN no config file, WHEN `/ship` runs, THEN shipped defaults apply and the run completes successfully.
- GIVEN `SPEQ_STEPS_IMPLEMENT_MODEL=gpt-4o-mini`, WHEN `/implement` runs, THEN that model is used regardless of config.
- GIVEN `--set steps.spec.prompt_version=v2`, WHEN `/ship` runs, THEN `speq sync-prompts` materializes `prompts/spec/v2.md` into `.claude/commands/spec.md` before `/spec` executes.
- GIVEN a step marked `skip_if: capability >= verify-native` and a capable configured model, WHEN `/ship` runs, THEN that step is elided and the elision is recorded in the run log.
- GIVEN `/project:config`, WHEN run interactively, THEN current settings are displayed and edits are validated before writing.
- GIVEN a `CLAUDE.md` with a `<!-- BEGIN SPEQ -->` block containing pipeline keys, WHEN `/project:config migrate` runs, THEN those keys are lifted into `speq.config.yaml` and removed from `CLAUDE.md`; non-pipeline content is preserved.

**Non-negotiables:**

- *Security.* Config schema rejects fields named `*_secret` or `*_token`; secrets must come from env.
- *Failure.* Invalid config produces a friendly error before any step runs; never partial-executes.
- *Idempotency.* Re-running with the same config produces equivalent artifacts.
- *Performance.* Config load completes in <100ms.

### F2: ADRs (generation + guardrails)

Add `/project:adr` command family that captures architecture decisions as machine-readable rubrics enforced through the pipeline.

**Resolved: Markdown entry points + TS handlers.** The slash-command surface stays consistent (markdown in `.claude/commands/`, Claude Code's native discovery). The body of each prompt shells out to TS handlers for deterministic work. General rule: deterministic work goes in TS, inference-driven work stays in markdown prompts. Per command:
- `/project:config` — thin markdown shim; TS handler does YAML read/validate/write. Interactive UX via Claude reasoning over the handler's output.
- `/project:adr-bootstrap` — Claude-driven prompt for the interview, with TS helpers for structural scan (parsing manifests, walking folder structure) and ADR file writes.
- `/project:adr add | supersede | list` — thin shims; TS handlers do file ops.
- `/project:board` — one-line shim invoking a TS renderer; HTML generation is deterministic.
- `/project:eval-workflow` — markdown prompt orchestrates Claude-graded scoring; TS helpers run `/ship` in subprocesses and capture output.
- `/project:eval-step` — similar mix (per-step grader runner).
The spec phase will decide the exact split per command.

ADRs live as `docs/adr/NNNN-slug.md` by default (industry convention; discoverable by people not using speq, compatible with `adr-tools` and similar). The path is overridable via `speq.config.yaml.adr.path` for projects with an established alternate convention. Speq-internal artifacts (run logs, evals, board HTML) stay under `.speq/`; ADRs live in `docs/` as project artifacts.

Frontmatter: `id`, `status` (`draft|active|superseded`), `supersedes` (optional), `tags`, plus body sections for Context, Decision, Consequences.

**Generation for existing codebases — hybrid approach:**

`/project:adr-bootstrap` runs three passes:

1. **Structural scan.** Walks current repo state and infers decisions from evidence: dependency manifests (`package.json`, `requirements.txt`, `Cargo.toml`, `go.mod`), config files (auth provider, deployment target, ORM, queue), folder structure (monorepo, layered, hexagonal), framework conventions. Fast, high signal, focuses on present truth.
2. **Docs ingestion.** Parses `README.md`, `ARCHITECTURE.md`, `CONTRIBUTING.md`, top-level code comments, RFC directories. Captures whatever the team already wrote down.
3. **Interactive interview.** For low-confidence drafts, surfaces clarifying questions to the user. "I see Postgres + Drizzle — was this chosen over alternatives, or template default?"

Optional `--include-history` flag adds a fourth pass: **commit-epoch clustering.** Groups commits by changed-file scope, identifies periods where major shifts happened (new top-level modules, dependency adds, framework swaps), generates one ADR per epoch boundary. Useful for repos with rich commit history but secondary to structural scan — commit-by-commit is too noisy.

All generated ADRs start `status: draft`. Only user-approved ADRs gate the pipeline. PR/issue mining is a future-scope extension.

**Enforcement through the pipeline:**

- `/requirements`: ADR set is part of codebase scan; surfaces relevant ADRs during PRD gathering ("ADR-0007 says no Redis — note for this PRD?").
- `/enrich`: PRD claims cross-referenced against active ADRs; violations flagged for resolution before PRD is locked.
- `/plan`: validates that decomposition does not require an ADR violation; flags impossible tasks.
- `/verify`: adds an eighth gate — "diff honors active ADRs"; tooling-checked where possible (e.g., dependency add against a "no Redis" ADR), prose-checked otherwise.

**Supersession:** `/project:adr supersede N` is the only path to retire an active ADR. Creates a successor with `supersedes: N`, marks N `superseded`, and only the successor gates from that point on. No silent violation.

**Acceptance:**

- GIVEN an existing codebase, WHEN `/project:adr-bootstrap` runs, THEN draft ADRs are generated from structural scan + docs and the user approves each before it becomes active.
- GIVEN active ADRs and a PRD whose enriched form would violate one, WHEN `/enrich` runs, THEN the violation is surfaced and the run stops for user resolution.
- GIVEN active ADRs and a `/verify` run whose diff violates one, WHEN the gate evaluates, THEN it fails with the violating ADR cited by id.
- GIVEN `/project:adr supersede N`, WHEN executed, THEN ADR N is marked `superseded`, the new ADR's `supersedes: N` field is set, and only the new ADR gates subsequent runs.

**Non-negotiables:**

- *Security.* ADRs can encode security invariants (e.g., "all PII at rest is encrypted with KMS-managed keys"); `/verify` gate must catch violations.
- *Failure.* Bootstrap failures preserve any drafts already produced; never lose user-confirmed ADRs on retry.
- *Idempotency.* Re-running `/project:adr-bootstrap` on a repo with existing active ADRs only adds drafts for newly observed decisions.
- *Performance.* ADR scan during `/requirements`, `/enrich`, `/verify` completes in <2s for repos with <50 ADRs.

### F3: Evals

Turn `/verify` from prose-judgment into a regression-tested layer. Two eval surfaces:

**Workflow evals.** Fixture repos checked into `.speq/evals/fixtures/` by default (visible, version-controlled, easy to inspect); the path is overridable via `speq.config.yaml.evals.fixtures_path` for projects that want them out of `.git`. Each fixture is a minimal PRD + target codebase + golden expected output. Size discipline: <1MB per fixture. A `speq eval fixture add <path>` helper copies a fixture in while excluding `.git`, `node_modules`, `target/`, `dist/`, and similar bulk. `/project:eval-workflow` replays `/ship` against each fixture and reports diff vs. golden. Suitable for nightly/CI automation later.

**Step evals.** Each step ships a rubric. A small Haiku-backed grader scores the step's output against the rubric — e.g., "PRD covers security? failure? idempotency? performance?" produces a numeric score plus qualitative report.

**Eval registry.** Each step declares which rubric version is active in `speq.config.yaml`. Bumping a prompt version, swapping an implementation, or routing to a different model triggers step evals automatically as a gate for accepting the change.

**Acceptance:**

- GIVEN a fixture repo and a prompt change, WHEN `/project:eval-workflow` runs, THEN regression is detected (diff vs. golden reported with severity).
- GIVEN a step output, WHEN the step-eval grader runs, THEN a numeric score (0–100) and qualitative report are produced.
- GIVEN a proposed `speq.config.yaml` change, WHEN saved via `/project:config`, THEN affected step evals run and the delta vs. baseline is reported before commit.
- GIVEN an eval run, WHEN complete, THEN results are written to `.speq/evals/runs/<timestamp>/` for historical comparison.

**Non-negotiables:**

- *Security.* Eval fixtures must not contain real credentials; placeholder values only.
- *Failure.* A failing fixture does not block other fixtures; failures aggregate into the run report.
- *Idempotency.* Same fixture + same config → equivalent scores (within grader noise tolerance).
- *Performance.* Single-fixture workflow eval completes in <5min; step eval completes in <30s.

### F4: Cost telemetry

Instrument every step to emit per-run cost data, surface it on the sprint board, and use it to gate expensive runs.

**Per-step records** written to `.speq/runs/<run-id>/<step>.json` containing: model used, tokens in, tokens out, wall clock, retry count, tool call count, status, and a `source` field (`exact|estimated`). Records contain **metadata only — no prompt or output content.**

**Usage capture is interface-shaped, with two implementations.** *Exact* reads Claude Code's per-turn token data via whatever surface that turns out to be (hook output, log file, or CLI command — to be probed at install time). *Estimated* falls back to a character-count / 4 heuristic on prompt + output. Speq probes the exact path at install; on success defaults to exact, otherwise falls back to estimated and surfaces "costs are approximate" on the board. The downstream rollup, prediction, and guardrail logic is identical for both paths — only the data source differs.

**Roll-ups** per run (`.speq/runs/<run-id>/summary.json`) and per command. The sprint board (F6) displays current run cost and historical comparison.

**Predictive estimation:**

- *Cold start* (no history): rough per-command averages from a shipped model card (`$/1M tokens` per model).
- *After N≥5 historical runs in the project:* fit a simple regression on `(repo size, PRD size, spec count) → per-step usage`. Report estimate with confidence band.

**Guardrail (v2 feature, behind flag):** `/ship --max-cost $X` halts before any step that would push predicted total over X, prompting the user to confirm. The board shows live cost delta when `speq.config.yaml` changes (e.g., "switching `/implement` to Codex drops this run from $15 to $5").

**Acceptance:**

- GIVEN a `/ship` run, WHEN each step completes (success or failure), THEN a per-step usage record is written.
- GIVEN the exact-path probe succeeded at install, WHEN a step completes, THEN the record's `source` field is `exact`.
- GIVEN the exact-path probe failed at install, WHEN a step completes, THEN the record's `source` field is `estimated` and the board surfaces a note that costs are approximate.
- GIVEN N≥5 historical runs in the project, WHEN `/ship` starts, THEN a predicted cost band is displayed.
- GIVEN `--max-cost X` and predicted cost > X, WHEN `/ship` runs, THEN user is prompted to confirm before the offending step.
- GIVEN a `speq.config.yaml` edit that changes a step's model, WHEN saved, THEN cost prediction is recomputed and shown.

**Non-negotiables:**

- *Security.* Usage records contain no prompt or output content; only metadata and aggregate sizes.
- *Failure.* A missing or malformed record on one step does not break rollups for others; degrade to "unknown" for that step.
- *Idempotency.* Re-running a step overwrites that step's record, not the run's other records.
- *Performance.* Writing a usage record adds <50ms to step latency.

### F5: Per-step model routing

Build on F1's config: each step in `speq.config.yaml` carries a `model` field. The field accepts either a **family shorthand** (`haiku`, `sonnet`, `opus`) or an **exact snapshot ID** (`claude-sonnet-4-7`, `claude-haiku-4-5-20251001`). Shorthands resolve at config-load time via a shipped *model card* (`models.yaml` in the speq repo) that maps each family to its current snapshot. Shorthand tracks "latest"; pin to an exact ID for reproducibility. The model card is bumped when Anthropic ships new models. Non-Claude models (e.g., `gpt-4o-mini`) are also valid values; the step's implementation is responsible for routing to the correct CLI (e.g., the implementer shells out to `codex exec` when a `gpt-*` model is requested).

Shipped defaults (all shorthands):

| Step | Default model | Rationale |
|---|---|---|
| `/requirements` scan | haiku | Mechanical codebase scan |
| `/enrich` scan | haiku | Same |
| `/spec` | haiku | PRD → OpenSpec, mostly mechanical |
| `/plan` | sonnet | Decomposition is judgment-heavy |
| `/implement` | sonnet | Configurable to gpt-4o-mini via codex CLI for ~3x savings |
| `/verify` | sonnet | Judge work |
| `/done` | haiku | Reconciliation, mostly mechanical |

Additionally, `/plan` tags each Beads issue with a `complexity` label (`trivial|simple|standard|complex`). `/implement` consults the label and routes per-task: trivial/simple → cheap model, standard/complex → primary model.

**Resolved: Defaults in models.yaml, overridable per-project.** Shipped defaults in `models.yaml`:
```yaml
complexity_routing:
  trivial: haiku
  simple: haiku
  standard: sonnet
  complex: sonnet
```
`speq.config.yaml.steps.implement.complexity_routing` overrides per-project, merged key-by-key (change just `complex` to `opus` without restating the others). Consistent with F1's defaults-plus-override pattern.

**Acceptance:**

- GIVEN `steps.spec.model: haiku` in config, WHEN `/spec` runs, THEN the model card resolves `haiku` to its current snapshot and that model is used.
- GIVEN `steps.spec.model: claude-haiku-4-5-20251001` in config, WHEN `/spec` runs, THEN that exact snapshot is used regardless of model card state.
- GIVEN `steps.implement.model: gpt-4o-mini` in config and codex CLI installed, WHEN `/implement` runs, THEN the implementer shells out to `codex exec` with that model.
**Resolved: Runner abstraction.** Define a `Runner` interface: `{ name, supports(model) → bool, exec(prompt, model, opts) → {output, usage} }`. Ship two implementations:
- **ClaudeRunner** — current behavior, shells out to `claude` with current arguments. Supports `haiku`/`sonnet`/`opus`/`claude-*`.
- **CodexRunner** — shells out to `codex exec` with codex's argument format. Supports `gpt-*`.

`runPhase()` resolves the model (via shorthand → `models.yaml`), then dispatches to the runner whose `supports()` returns true. CLI argument differences are isolated inside each runner; `runPhase()` is runner-agnostic.

`models.yaml` tags each entry with its runner so lookup is data-driven:
```yaml
models:
  haiku: { snapshot: claude-haiku-4-5-20251001, runner: claude }
  sonnet: { snapshot: claude-sonnet-4-7, runner: claude }
  gpt-4o-mini: { runner: codex }
```
Optional escape hatch: `speq.config.yaml.steps.<name>.runner: codex` lets users force a runner override. Adding a future runner (e.g., Ollama) is just a new file in `src/runners/` and a new entry in `models.yaml` — no changes to `runPhase()`.
- GIVEN `/plan` assigns complexity labels to issues, WHEN `/implement` processes each issue, THEN the per-issue model is selected from the complexity-to-model map in config.
- GIVEN a step with the default model and no override, WHEN it runs, THEN the shipped default applies.

**Non-negotiables:**

- *Security.* Model selection happens within the slash command; never exposed in artifacts.
- *Failure.* Requested model unavailable → falls back to a documented next-best with a warning logged.
- *Idempotency.* Same config + same input → same model selection.
- *Performance.* Model selection adds <10ms to step start.

### F6: Live sprint board

`/project:board` renders current run + Beads state + cost data as a self-contained HTML file at `.speq/board.html`. Includes a manual refresh button (re-runs the command).

**Surface:**

- **Pipeline lanes:** PRD → Specs → Plan → Implement → Verify → Done. Current step highlighted.
- **Cards per Beads issue:** status, complexity label, assignee (if multi-agent), test status, links to spec/test/diff artifacts.
- **Configuration panel:** which implementation, prompt version, and model is configured per step.
- **Cost panel:** tokens used so far, predicted total with confidence band, wall clock.
- **Run history:** past N runs with timestamp, outcome, cost.

v1 is manual refresh. A file-watching `speq daemon` for live updates is deferred to v2.

**Acceptance:**

- GIVEN a running `/ship` and a refresh of `board.html`, WHEN the page loads, THEN current step, Beads state, and cost-to-date are shown.
- GIVEN `speq.config.yaml` edited, WHEN the board reloads, THEN the configuration panel reflects the change.
- GIVEN a Beads issue with a complexity label, WHEN the card renders, THEN the label is visible.

**Non-negotiables:**

- *Security.* `board.html` is self-contained (no external resources beyond optional CDN-loaded styling); contains no secrets.
- *Failure.* If Beads state can't be read, the board renders what it can and surfaces the failure inline.
- *Idempotency.* Rendering twice on the same state produces identical HTML.
- *Performance.* Render completes in <2s for projects with <100 Beads issues.

### F7: Multi-agent `/implement` (opt-in)

When `steps.implement.impl: agent-teams` is configured, `/implement` hands off to [agent-teams'](https://github.com/sransom/agent-teams) `/spawn` with the Beads task graph produced by `/plan`. agent-teams handles explorer → parallel implementer worktrees → judge → reviewers → integrate.

`/verify` runs after integrate completes, against the merged diff.

When `steps.implement.impl: sequential` (default), the current single-agent loop runs unchanged.

**Acceptance:**

- GIVEN `steps.implement.impl: agent-teams` in config and agent-teams installed, WHEN `/implement` runs, THEN `agent-teams /spawn` is invoked with the Beads graph and the configured implementer model.
- GIVEN passing fan-out arms, WHEN integrate completes, THEN `/verify` runs against the merged diff.
- GIVEN agent-teams not installed and that impl configured, WHEN `/implement` runs, THEN a clear error is surfaced with installation instructions.

**Non-negotiables:**

- *Security.* Each worktree honors the project's secret-management posture (env-only); no secrets in worktree state.
- *Failure.* A failing arm reopens its Beads issue and triggers retry (up to 3, per agent-teams convention); escalation to user after.
- *Idempotency.* Re-running with the same Beads state resumes via `/spawn --resume`.
- *Performance.* Orchestration overhead beyond agent-teams' own is <5s.

### F8: Durable orchestrator state via thrum (single-user)

Install [thrum](https://leonletto.github.io/thrum/) as an optional dependency. When configured, speq writes orchestrator decision-log messages to a local thrum profile during `/plan`, `/implement`, and `/verify`. The log captures decisions made in flight that aren't yet reflected in code or specs (e.g., "skipped task T-12 because dependency T-09 changed signature; rerouted T-15").

**Consumers:**

- `/done` reads the thrum log to populate the PR description with rationale.
- `/ship --resume` reads thrum state in addition to Beads to recover mid-step context.

Single-user, single-machine scope — thrum's multi-machine sync features are deferred.

**Acceptance:**

- GIVEN a `/ship` interrupted mid-`/implement`, WHEN `/ship --resume` runs, THEN orchestrator decisions made before the interruption are recovered from thrum and reflected in the resumed run.
- GIVEN `/done` at run end, WHEN the PR description is generated, THEN it includes a "Decisions" section sourced from the thrum log.
- GIVEN thrum not installed, WHEN any speq command runs, THEN the command runs unchanged (graceful degrade).

**Non-negotiables:**

- *Security.* thrum messages contain rationale only; no secrets, no credentials.
- *Failure.* thrum unavailable → falls back to in-process scratchpad with a warning logged.
- *Idempotency.* Re-writing the same decision is a no-op.
- *Performance.* Decision-log writes add <20ms to step latency.

## Security & Compliance

### Threat Model

| Threat | Likelihood | Impact | Mitigation |
|--------|-----------|--------|------------|
| **Command injection via config values** | Medium | High | `runner.ts` already uses `spawn` with `shell: false`. F1 config values must never be interpolated into shell strings. Config schema validation rejects suspicious patterns. |
| **Secret leakage into config/artifacts** | Medium | High | Config schema rejects `*_secret`/`*_token` fields. `.speq/runs/` records contain token counts only, never prompt content. Eval fixtures are validated for placeholder-only credentials. |
| **Path traversal in prompt versioning** | Low | High | `speq sync-prompts` must validate that resolved prompt paths are within `prompts/` directory. Reject `../` sequences and symlinks outside the repo. |
| **Malicious eval fixtures** | Low | Medium | `speq eval fixture add` strips `.git`, `node_modules`, etc. Fixtures run in the project's own sandbox. Size limit enforced (<1MB per fixture). |
| **ADR bypass / silent violation** | Medium | Medium | `/verify` gate fails hard on ADR violations. Only `/project:adr supersede` can retire an active ADR. No runtime override. |
| **Cost telemetry data exfiltration** | Low | Low | `.speq/` is local-only, no network calls. `.gitignore` should include `.speq/runs/` to prevent accidental commit of usage data. |
| **Dependency confusion (agent-teams, thrum)** | Low | High | F7 and F8 are opt-in. Install instructions must specify exact package sources. `runCommand` validates binary exists via `which` before invocation. |

### Data Classification

| Data | Sensitivity | Location | Retention |
|------|------------|----------|-----------|
| `speq.config.yaml` | Internal | Repo root (committed) | Permanent |
| `models.yaml` | Public | Speq package | Permanent |
| `docs/adr/*.md` | Internal | Repo (committed) | Permanent |
| `.speq/runs/*/` (cost records) | Internal | Local only | Prunable; last 100 runs |
| `.speq/evals/fixtures/` | Internal | Repo (committed) | Permanent |
| `.speq/evals/runs/` | Internal | Local only | Prunable |
| `.speq/board.html` | Internal | Local only | Regenerated on demand |
| Thrum decision log | Internal | Local only (thrum profile) | Session-scoped |

### Authentication & Authorization

Single-user, single-machine. No auth layer. Pipeline trusts the local filesystem and the user's shell environment. External CLI tools (`claude`, `bd`, `codex`, `agent-teams`, `thrum`) authenticate via their own mechanisms (API keys in env vars). Speq never handles or proxies credentials.

## Operational Requirements

### Metrics to emit

Speq is a CLI tool, not a service. "Metrics" are written to `.speq/runs/` as JSON files, not to a metrics backend.

- Per-step: model used, tokens in/out, wall clock, retry count, tool call count, status, source (exact|estimated) — F4
- Per-run: total tokens, total cost estimate, step count, pass/fail, duration — F4
- Per-eval: score (0–100), qualitative report, fixture name, config snapshot — F3
- Per-ADR-check: ADRs evaluated, violations found, gate pass/fail — F2

### Alarm conditions

Not applicable (single-user CLI). The sprint board (F6) surfaces:
- Cost overruns: predicted total exceeds `--max-cost` threshold (when enabled)
- Eval regressions: score delta exceeds configured threshold after a config change
- Stale runs: `.speq/runs/` exceeding 100 entries triggers a prune suggestion

### Runbook sketch

**Scenario 1: Pipeline fails mid-`/implement` (most common)**
1. Check error output in terminal.
2. Run `speq resume` — auto-detects state from artifacts + Beads + thrum (if available).
3. If resume fails, run `speq ship --from=implement` to restart from implement phase.
4. If a specific task is failing, use `bd show <id>` to inspect, then fix manually and `bd close <id>`.

**Scenario 2: Config change breaks pipeline**
1. Run `speq config` to display current settings.
2. Check `.speq/runs/<latest>/` for the failing step's JSON record.
3. Revert `speq.config.yaml` to last known-good (git checkout).
4. Re-run affected step evals before re-applying the change.

## Failure Modes

| Dependency | Failure scenario | Detection | Degraded behavior | Recovery |
|------------|-----------------|-----------|-------------------|----------|
| `claude` CLI | Not installed or not in PATH | `which` check in `runPhase()` (existing) | Hard fail with install instructions | User installs Claude Code CLI |
| `bd` (Beads) | Not installed or `.beads/` missing | `which` check + `existsSync` (existing) | Hard fail for plan/implement/verify; init offers to install | `npm install -g @beads/bd && bd init` |
| `codex` CLI (F5/F7) | Not installed when `gpt-*` model configured | `which` check before step dispatch | Hard fail with install instructions; suggest falling back to Claude model | User installs codex or changes model config |
| `agent-teams` (F7) | Not installed when `impl: agent-teams` configured | `which` check before `/implement` | Hard fail with install instructions; suggest `impl: sequential` fallback | User installs agent-teams or reverts config |
| `thrum` (F8) | Not installed when configured | Probe at install time / runtime check | Graceful degrade to in-process scratchpad; warning logged; `/ship --resume` still works via Beads artifacts | User installs thrum or continues without |
| `speq.config.yaml` (F1) | Missing, malformed, or schema-invalid | Schema validation at config load (<100ms) | Missing: shipped defaults apply. Malformed: hard fail with friendly parse error before any step runs | User fixes YAML or deletes to use defaults |
| `.claude/commands/*.md` (F1) | Missing after `sync-prompts` or init | `existsSync` check in `runPhase()` (existing) | Hard fail: "Run `speq init` first" | `speq init` or `speq sync-prompts` |
| `models.yaml` (F5) | Shipped file missing or corrupt | Read at config load | Hard fail: "speq installation corrupt" | Reinstall speq |
| Eval fixtures (F3) | Fixture missing expected files | Fixture validation before eval run | Skip fixture, aggregate failure into run report; other fixtures continue | Fix or regenerate fixture |
| Filesystem (`.speq/`) | Permission denied or disk full | Catch on write | Cost record / eval result lost for that step; pipeline continues; board may show stale data | Fix permissions or free disk space |
